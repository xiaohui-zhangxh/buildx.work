## 商业价值分析阶段 ⭐

> **重要**：在开始任何技术开发之前，必须先进行商业价值分析。站在投资人和用户的视角，评估项目的可行性、价值输出和功能优先级。

> **AI 执行方式**：参考 `.cursor/commands/70-business-analysis.md` 指令，AI 应该主动进行全面的商业价值分析，而不是等待用户提供信息或反馈。

### 1. 投资人视角：核心问题

#### 市场机会
- **市场规模**：目标市场有多大？是否足够大？
- **市场趋势**：市场是在增长还是萎缩？
- **竞争格局**：现有竞争对手有哪些？我们的差异化在哪里？

#### 用户需求
- **真实需求**：用户真的需要这个产品吗？
- **痛点强度**：用户痛点有多强烈？（1-5 分）
- **付费意愿**：用户愿意为这个产品付费吗？（1-5 分）

#### 商业模式
- **价值主张**：我们为用户创造什么价值？
- **盈利模式**：如何赚钱？（免费+增值、订阅、按量付费等）
- **成本结构**：主要成本是什么？（开发、运营、营销等）
- **单位经济**：单个用户的获取成本（CAC）和生命周期价值（LTV）

#### 竞争优势
- **技术壁垒**：有什么技术优势？
- **数据壁垒**：能否积累有价值的数据？
- **网络效应**：是否有网络效应？
- **品牌优势**：能否建立品牌认知？

#### 执行能力
- **团队能力**：团队是否有能力执行？
- **资源需求**：需要多少资源？
- **时间窗口**：市场窗口期有多长？

### 2. 用户视角：核心问题

#### 用户画像
- **目标用户**：谁会用这个产品？（具体描述）
- **用户场景**：用户在什么场景下使用？
- **使用频率**：用户多久用一次？（每日、每周、每月）

#### 价值输出
- **核心价值**：用户使用后能得到什么？
- **情感价值**：用户的情感需求是什么？（成就感、归属感等）
- **功能价值**：用户的功能需求是什么？

#### 使用体验
- **易用性**：产品是否容易上手？
- **效率提升**：能否提升用户效率？
- **愉悦感**：使用过程是否愉悦？

#### 付费意愿
- **免费功能**：哪些功能应该免费？
- **付费功能**：哪些功能值得付费？
- **价格敏感度**：用户对价格有多敏感？

### 3. 功能优先级评估

**评估标准**：
- **用户价值**：功能对用户的价值（1-5 分）
- **商业价值**：功能对商业的价值（1-5 分）
- **开发成本**：开发难度和成本（1-5 分）
- **优先级**：P0（必须）、P1（重要）、P2（可选）、P3（未来）

**输出**：
- 功能优先级列表
- 功能价值评估表
- MVP 功能清单

### 4. 风险评估

**需要识别**：
- 市场风险
- 技术风险
- 商业风险
- 执行风险

**输出**：
- 风险清单
- 应对策略

### 5. 商业价值分析文档

创建 `BUSINESS_ANALYSIS.md` 文档，包含：
- 市场机会分析
- 用户需求分析
- 商业模式设计
- 竞争优势分析
- 功能优先级评估
- 风险评估
- 关键成功因素

**参考模板**：创建新项目时，可以参考已完成项目的 `BUSINESS_ANALYSIS.md` 文档作为模板

## 项目规划阶段

**核心问题**：
- 项目要解决什么问题？

**输出**：
- 项目定位说明

## 项目命名阶段

### 1. 命名原则

**核心要求**：
- ✅ **唯一性**：避免与现有品牌重名
- ✅ **SEO 友好**：有利于搜索引擎搜索
- ✅ **域名可用**：优先考虑 .com 域名
- ✅ **易记易拼**：用户容易记住和拼写
- ✅ **品牌感**：有独特的品牌识别度

### 2. 命名策略

#### 策略 1：组合词
- **方法**：核心词 + 动作词/工具词/创意词
- **示例**：Task + Forge = TaskForge（任务锻造）、Note + Hub = NoteHub（笔记中心）
- **优势**：包含核心关键词，SEO 友好

#### 策略 2：变体字母
- **方法**：使用变体字母（K、X、Z 等）
- **示例**：Task → Tasko、Note → Notex
- **优势**：增加独特性，避免直接竞争

#### 策略 3：新造词
- **方法**：创造新词，但要有意义
- **示例**：Task + Syn = TaskSyn（任务合成）、Data + Mosa = DataMosa（数据马赛克）
- **优势**：唯一性强，易于建立品牌

#### 策略 4：后缀策略
- **方法**：使用独特后缀（-ify、-io、-ly 等）
- **示例**：Taskify（任务化）、Notely（笔记化）
- **优势**：现代感强，易于理解

### 3. 命名检查清单

#### ✅ 步骤 1：生成备选方案
- 生成 10-15 个备选名称
- 每个名称包含：英文名、中文名、核心特点
- 记录在文档中（如 `NAME_OPTIONS.md`）

#### ✅ 步骤 2：搜索引擎检查
- 在 Google 搜索每个名称
- **记录搜索结果数量**（越少越好）
- 检查是否有同名品牌或产品
- **评分标准**：
  - ⭐⭐⭐⭐⭐：搜索结果 < 100
  - ⭐⭐⭐⭐：搜索结果 100-1000
  - ⭐⭐⭐：搜索结果 1000-10000
  - ⭐⭐：搜索结果 > 10000

#### ✅ 步骤 3：域名可用性检查
- **优先检查 .com 域名**
- 使用域名查询工具：
  - GoDaddy：https://www.godaddy.com/zh-sg/domains
  - Namecheap：https://www.namecheap.com/
  - Whois 查询：https://www.whois.com/
- **记录域名状态**：
  - ✅ 可注册
  - ❌ 已被注册（记录注册时间）
  - ⚠️ 待售（可能可以购买）

#### ✅ 步骤 4：商标检查 ⭐ **最重要**

**为什么重要**：
- 商标侵权可能导致法律纠纷
- 商标冲突可能导致品牌无法注册
- 商标冲突可能影响产品推广

**检查方法**：

1. **中国商标查询**
   - **官方数据库**：中国商标网（http://sbj.cnipa.gov.cn/）
   - **查询步骤**：
     1. 访问中国商标网
     2. 选择"商标综合查询"或"商标近似查询"
     3. 输入要查询的名称
     4. 查看查询结果
   - **检查内容**：
     - 是否有相同或近似商标
     - 商标注册类别（特别是第 42 类：计算机软件、网站服务等）
     - 商标状态（已注册、申请中、已失效等）
     - 注册人信息

2. **美国商标查询**
   - **官方数据库**：USPTO（https://www.uspto.gov/trademarks/search）
   - **查询步骤**：
     1. 访问 USPTO 网站
     2. 使用 TESS（Trademark Electronic Search System）
     3. 输入要查询的名称
     4. 查看查询结果

3. **国际商标查询（可选）**
   - **WIPO 全球品牌数据库**：https://www3.wipo.int/branddb/en/

4. **商标分类检查**
   - **重点关注类别**：
     - 第 9 类：计算机软件、应用程序
     - 第 35 类：广告、商业管理
     - 第 42 类：计算机软件设计、网站服务
     - 第 45 类：在线社交网络服务

5. **商标近似度评估**
   - **评估标准**：
     - 完全相同：❌ 高风险，不能使用
     - 高度近似：⚠️ 高风险，不建议使用
     - 中度近似：⚠️ 中风险，需要评估
     - 低度近似：✅ 低风险，可以使用

**记录检查结果**：
- 中国商标状态（✅ 无冲突 / ⚠️ 有近似 / ❌ 有相同）
- 美国商标状态
- 商标注册类别
- 商标风险评估（⭐⭐⭐⭐⭐ 无风险 到 ⭐⭐ 高风险）

#### ✅ 步骤 5：社交媒体检查
- 检查主要社交媒体平台：
  - Twitter / X
  - Instagram
  - Facebook
  - GitHub
- 记录用户名可用性

#### ✅ 步骤 6：综合评估
- **评估标准**（按优先级排序）：
  1. **无商标冲突** ⭐ **最重要**：无商标冲突或低风险（优先）
  2. **唯一性**：搜索结果少（优先）
  3. **域名可用**：.com 域名可注册（优先）
  4. **品牌感**：有独特的品牌识别度
  5. **易记易拼**：用户容易记住和拼写
- **⚠️ 重要提醒**：
  - **商标冲突是最高风险**：即使其他条件都很好，如果有商标冲突，也不建议使用
  - **优先选择无商标冲突的方案**：即使搜索结果稍多，只要无商标冲突，也是更好的选择
- **排序推荐**：根据综合评估排序
- **最终确认**：选择无商标冲突、唯一性强且域名可用的名称

### 1. 项目文档结构

**标准文档结构**（使用编号前缀，方便按顺序阅读）：

```
docs/
└── project-name/
    ├── README.md                    # 文档索引（不使用数字前缀）
    ├── 01-BUSINESS_ANALYSIS.md      # 商业价值分析 ⭐ 第一步
    ├── 02-PROJECT_PLAN.md           # 项目规划
    ├── 03-PRODUCT_DESIGN.md         # 产品设计 ⭐ 重要

    ├── 05-DATA_MODELS.md            # 数据模型设计
    ├── 06-AI_INTEGRATION.md         # AI 集成方案（如有）
    ├── 07-DEVELOPMENT_PLAN.md       # 开发计划
    └── phase-*/                     # 各阶段详细文档
        ├── plan.md
        ├── progress.md
        └── notes.md
```

**文档编号规范**：
- README.md 不使用数字前缀（直接使用 `README.md`）

- 编号后加连字符 `-`
- 文件名使用大写字母（如 `BUSINESS_ANALYSIS.md`）
- 编号顺序：README（无前缀）→ 商业价值分析（01）→ 项目规划（02）→ 产品设计（03）→ ...

## 项目初始化阶段

### 1. 基于 BuildX.work 创建项目

**步骤**：

```bash
# 1. Fork/Clone BuildX.work
git clone <buildx.work-repo> project-name
cd project-name

# 2. 初始化 Git（如果需要独立仓库）
rm -rf .git
git init
git add .
git commit -m "Initial commit from buildx.work"

# 3. 添加上游仓库（用于后续更新）
git remote add upstream <buildx.work-repo>

# 4. 安装依赖
bundle install
npm install

# 5. 设置数据库
bin/rails db:setup

# 6. 启动开发服务器
bin/dev
```

### 2. 项目配置

**需要配置**：
- 项目名称
  - 业务项目的文档应该放在 `docs/project-[项目名称]/` 目录下
  - **更新根目录 README**：在根目录 `README.md` 的"当前业务项目"部分，更新项目名称和链接地址
    - 找到 `README.md` 中的"当前业务项目"部分
    - 更新项目名称和链接地址，格式：`📋 [项目显示名称](docs/project-[项目名称]/README.md) - 项目文档索引`
    - **示例**：
      - 如果项目目录名为 `my-app`，项目显示名称为 `MyApp`
      - 则更新为：`📋 [MyApp](docs/project-my-app/README.md) - 项目文档索引`
    - **注意**：只更新业务项目链接部分，不要修改其他基础设施文档内容
- 环境变量（.env 文件）

## 经验总结

### 商业价值分析经验
1. **先分析再开发**：在写代码之前，先进行商业价值分析
2. **用户视角优先**：始终站在用户角度思考问题
3. **数据驱动**：用数据验证假设，而不是凭感觉
4. **功能优先级**：根据用户价值和商业价值确定功能优先级
5. **成本意识**：始终考虑成本，特别是运营成本（如 AI API 成本）

### 命名经验
1. **唯一性优先**：搜索结果少比好听更重要
2. **域名检查**：.com 域名可用性很关键
3. **组合词策略**：核心词 + 独特词组合效果好
4. **变体字母**：使用 K、X、Z 等增加独特性

### 文档经验
1. **文档先行**：先规划再开发
2. **及时更新**：开发过程中及时更新文档
3. **结构清晰**：使用标准文档结构
4. **记录决策**：重要决策都要记录

### 技术选型经验
1. **基于 BuildX.work**：充分利用基础设施
2. **统一标准**：AI 功能统一使用 RubyLLM
3. **评估需求**：根据实际需求选择技术
4. **记录决策**：技术选择要有理由
